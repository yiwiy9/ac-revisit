# Implementation Plan

- [x] 1. 配布基盤と品質ゲートを整備する
- [x] 1.1 userscript 配布物の公開契約を成立させる
  - Greasy Fork へ公開できる metadata を持つ単一配布物を生成し、AtCoder 全体を対象 URL に固定する。
  - Tampermonkey 前提の権限宣言を最小限に留め、外部 JavaScript 読み込みを前提にしない。
  - 配布物のバージョン供給元を一元化し、必須 metadata が欠ける場合は生成を失敗させる。
  - _Requirements: 1.1, 1.2, 2.5_

- [x] 1.2 TypeScript・lint・型検査の独立した品質ゲートを用意する
  - 実装言語を TypeScript に固定し、ビルドとは別に型検査だけを実行できるようにする。
  - 静的解析を独立コマンドで実行できるようにし、公開前にまとめて検証できる入口を用意する。
  - userscript 用 API の型は既存の型定義に統一し、`any` への退避を避ける。
  - _Requirements: 1.3, 1.4, 1.5_

- [x] 2. 共通ルールと保存ワークスペースを固める
- [x] 2.1 ローカル日付の正規化と 14 日ルールを共通化する
  - ローカル暦日を一貫した日付キーへ正規化し、日次判定と比較に使う入口を一つにする。
  - 14 日経過判定と同日判定を共通ルールとして扱い、 ad-hoc な日付計算を分散させない。
  - 復習間隔は固定 14 日のみを扱い、動的調整や派生ルールを追加しない。
  - _Requirements: 4.1, 4.2, 7.8_

- [x] 2.2 単一スナップショット保存と復旧方針を実装する
  - 復習対象一覧と今日の一問状態を、単一の保存スナップショットとして読み書きする。
  - 保存値には問題識別子、表示用タイトル、登録日だけを保持し、禁止された追加属性は持たせない。
  - 保存形式や論理整合性が崩れている場合は部分復旧せず空状態へフォールバックする。
  - ストレージ読み書き失敗時は再試行せず、その操作だけを中止する。
  - 同一ブラウザプロファイルを単一アカウント前提で扱い、保存データをアカウント別に分離しない。
  - 複数タブ競合時は最後に成功した保存結果を採用し、厳密な競合解決は行わない。
  - _Requirements: 2.4, 2.6, 2.7, 2.9, 2.10, 2.11, 4.7, 7.4, 7.5, 7.16_

- [ ] 3. AtCoder 画面への導線差し込みを成立させる
- [ ] 3.1 ログイン判定と常設導線の差し込み条件を実装する
  - 確認済みヘッダー DOM シグナルだけでログイン状態を判定し、判定できない場合は未ログインとして扱う。
  - ログイン時のみ、ユーザーメニューに「ac-revisit 操作」を差し込む。
  - 提案候補の有無にかかわらず常設リンクは表示を試み、アンカーを解決できないページでは fail closed で省略する。
  - DOM 変更時の自動回復や継続監視は持たず、確認済み DOM のみを MVP 対象に限定する。
  - _Requirements: 2.1, 2.3, 2.8, 5.1, 5.2, 5.6, 7.11, 7.12, 7.13, 7.15_

- [ ] 3.2 問題文脈の解決と復習トグルの差し込みを実装する
  - 問題ページと提出詳細ページの両方で、現在の問題を特定できる場合にのみ復習トグルを表示する。
  - 未ログイン時はトグルを表示せず、ログイン済みでも問題文脈や差し込み位置を解決できない場合は表示しない。
  - 問題識別子と表示用タイトルは、確認済み DOM から安定して取り出せる情報だけを採用する。
  - 継続監視や自動再試行を追加せず、その時点の DOM 解決結果だけで表示可否を決める。
  - _Requirements: 2.2, 3.1, 3.2, 3.6, 7.12, 7.15_

- [ ] 4. 復習状態の更新ロジックを実装する
- [ ] 4.1 登録・解除・完了の単一トランザクションを実装する
  - 未登録の問題は当日を登録日として追加し、登録済みの再登録は現在状態を壊さず扱う。
  - 解除は復習対象から完全削除し、提案中の問題を解除した場合は同じ保存で今日の一問を完了状態へ遷移させる。
  - 完了操作は削除と当日での再登録を同一更新で行い、問題ごとの完了フラグは導入しない。
  - 完了判定は利用者の明示操作だけで進め、自動定着判定は行わない。
  - _Requirements: 3.3, 3.4, 3.5, 6.5, 7.9, 7.10_

- [ ] 4.2 当日初回の提案確定と日次消費を実装する
  - その日の最初の提案処理でだけ日次判定を消費し、候補があれば 1 問だけ選んで今日の提案へ設定する。
  - 前日の未完了提案は持ち越さず、候補がある場合は置き換え、候補がない場合は当日提案なしに更新する。
  - 常に同時に扱う提案問題は 1 問だけに保ち、追加の提案状態は持たない。
  - _Requirements: 4.3, 4.5, 4.6, 7.16_

- [ ] 4.3 自動通知の当日初回判定と抑制条件を実装する
  - ページロード時の自動起動がその日の最初の提案処理だった場合のみ、自動表示対象として扱う。
  - 候補がない日は常設リンク以外の追加通知を自動表示しない。
  - ストリークのような派生状態を導入せず、通知可否は当日提案確定結果だけで決める。
  - _Requirements: 4.4, 5.5, 7.6_

- [ ] 4.4 完了後の「もう一問」再抽選ルールを実装する
  - 「もう一問」は今日の一問が完了済みのときだけ有効にし、due 候補から 1 問だけ再抽選する。
  - 未完了の今日の一問を差し替える用途には使わせず、禁止状態は更新処理側でも拒否する。
  - 再抽選後は今日の一問を未完了状態へ戻し、同日の日次消費状態は維持する。
  - _Requirements: 6.8, 6.9, 7.8_

- [ ] 5. ポップアップ表示と操作整合を実装する
- [ ] 5.1 共通ポップアップ UI の表示骨格を実装する
  - 常設リンク経由と自動通知経由で同一のポップアップ UI を表示する。
  - 固定見出し「今日の一問」を常に表示し、一覧画面や統計表示のような追加セクションは導入しない。
  - 更新操作は単一ボタンに集約し、状態に応じた再描画へつなげられる構造にする。
  - _Requirements: 5.3, 5.4, 6.3, 7.2, 7.7_

- [ ] 5.2 問題リンクとアクションボタンの状態連動表示を実装する
  - 今日の一問が未完了かつ提案ありのときだけ問題タイトルを有効リンクにする。
  - 更新操作は状態に応じて「完了」または「もう一問」を表示する。
  - 今日の一問が完了済みで候補がない場合は「もう一問」を押下不可にし、デュー可視化は追加しない。
  - _Requirements: 6.1, 6.2, 6.4, 6.6, 6.7, 7.3_

- [ ] 5.3 ポップアップ操作前の整合確認と無言再描画を実装する
  - 問題タイトルリンクと更新ボタンの操作直前に、その時点の日付と最新保存状態との整合を確認する。
  - 同日不一致や日跨ぎが見つかった場合は、追加エラーメッセージを出さず、必要に応じて当日提案確定を伴う再同期後の状態へ再描画する。
  - ポップアップ表示中の継続監視は行わず、表示時と明示操作時だけ再検証する。
  - 直前の表示状態を使い回して不正な遷移を起こさないよう、 stale 状態を一箇所で収束させる。
  - _Requirements: 6.10, 6.11, 7.14_

- [ ] 6. 起動シーケンスと失敗時挙動を統合する
- [ ] 6.1 ページロード時の起動順序を一本化する
  - ページロード時は、ログイン判定、ページ種別判定、導線差し込み、当日提案確定、自動ポップアップ判定の順で一度だけ実行する。
  - 未ログインと判定した場合は、その時点で後続の UI 生成を行わず終了する。
  - 外部 API や外部サーバーとの通信は追加せず、ブラウザ内だけで処理を完結させる。
  - 自動 AC 検知や対象外画面向けの追加機能は起動時に生成しない。
  - _Requirements: 2.5, 2.8, 4.3, 4.4, 5.4, 7.1_

- [ ] 6.2 fail closed と開発時診断の扱いを統一する
  - DOM 差し込み不能、問題文脈未解決、ストレージ失敗を、利用者向け通知を増やさず fail closed で処理する。
  - 開発時だけ理由コード付きの診断情報を記録できるようにし、通常利用では恒常ログ出力にしない。
  - 診断情報は、発生箇所と操作種別を追える最小限の構造で扱う。
  - DOM 変更時は self-healing を増やさず、固定セレクタの追随修正で対応する前提を維持する。
  - _Requirements: 2.10, 3.6, 5.6, 7.12, 7.13_

- [ ] 7. 実装を検証し、MVP の受け入れ条件を固める
- [ ] 7.1 日付・保存・状態遷移の単体テストを整備する
  - 14 日判定、空状態フォールバック、登録・解除・完了・再抽選の更新結果を検証する。
  - stale 状態の拒否、候補なし時の日次消費、last-write-wins 前提の保存結果を確認する。
  - 禁止フィールドを保存しないことと、完了が削除→再登録で表現されることを確認する。
  - _Requirements: 2.4, 2.6, 2.7, 2.9, 2.10, 3.3, 3.4, 3.5, 4.1, 4.2, 4.3, 4.5, 4.7, 6.5, 6.8, 6.9, 7.8, 7.9, 7.16_

- [ ] 7.2 画面導線とポップアップの統合テストを整備する
  - ログイン時と未ログイン時の導線表示差、問題ページと提出詳細ページのトグル差し込みを確認する。
  - 常設リンクと自動通知が同一 UI を使うこと、候補なし時の表示制約、操作前整合確認後の再描画を確認する。
  - 対応外 DOM では静かに機能停止し、継続監視や余計な通知が発生しないことを確認する。
  - _Requirements: 2.1, 2.2, 2.3, 2.8, 3.1, 3.2, 3.6, 4.4, 5.1, 5.2, 5.3, 5.4, 5.5, 5.6, 6.1, 6.2, 6.3, 6.4, 6.6, 6.7, 6.10, 6.11, 7.11, 7.12, 7.14, 7.15_
