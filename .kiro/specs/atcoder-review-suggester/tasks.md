# Implementation Plan

- [x] 1. 配布契約と開発用品質ゲートを確定する
- [x] 1.1 userscript の配布契約を単一成果物として成立させる
  - Greasy Fork に投稿できる `.user.js` を唯一の配布物として生成し、Tampermonkey 前提の metadata を安定して含める。
  - AtCoder 全体を対象にする `@match` と、永続化に必要な最小限の `@grant` だけを公開契約として維持する。
  - 外部 JavaScript 参照や外部通信を持ち込まず、バージョン供給元を一元化して metadata の不整合は build failure として検出する。
  - _Requirements: 1.1, 1.2, 2.5_

- [x] 1.2 TypeScript 実装と独立した検証経路を維持する
  - 実装本体を TypeScript 前提で固定し、ビルドとは独立した型検査を常に実行できる状態にする。
  - ESLint を公開前の必須ゲートとして扱い、verify 系コマンドで lint・typecheck・test の順序を崩さず通せるようにする。
  - 開発用 build だけが復習間隔や診断境界を上書きできるようにし、公開物の既定仕様と混線させない。
  - _Requirements: 1.3, 1.4, 1.5_

- [x] 2. 共有基盤と保存ワークスペースの契約を固める
- [x] 2.1 ローカル日付ルールと候補抽出の共通基盤を実装する
  - ローカル暦日を canonical な日付キーへ正規化し、同日判定と経過日数計算の入口を一箇所へ集約する。
  - 登録日から 14 日以上経過した問題だけを候補として扱い、段階的増加や動的間隔調整を入れない。
  - 候補抽出とランダム選定は、日次提案と「もう一問」の両方から再利用できる共通ルールとして定義する。
  - _Requirements: 4.1, 4.2, 6.9, 7.8_

- [x] 2.2 review workspace の正規化と単一スナップショット保存を実装する
  - 復習対象と今日の一問状態をひとつの保存スナップショットとして読み書きし、問題ごとには識別子、表示用タイトル、登録日だけを保持する。
  - 前回完了日、問題別完了フラグ、習熟度、可変間隔、メモ、タグのような MVP 外フィールドは保存しない。
  - 保存形式や論理整合性が崩れている場合は部分復旧せず空状態へフォールバックし、読み書き失敗時は再試行せずその操作だけを中止する。
  - 同一ブラウザプロファイルを単一アカウント前提で扱い、アカウント別分離や厳密な競合解決を追加せず last-write-wins を許容する。
  - _Requirements: 2.4, 2.6, 2.7, 2.9, 2.10, 2.11, 4.7, 7.4, 7.5, 7.16_

- [x] 2.3 実行環境ポートと開発時診断の最小境界を整える
  - 乱数、userscript storage、開発時だけ有効な診断出力を、実行環境依存としてまとめて注入できるようにする。
  - 通常利用では恒常ログを出さず、開発時だけ理由コード、発生箇所、操作種別を追跡できるようにする。
  - 本番の振る舞いは fail-closed を優先し、診断のために仕様外の保存や表示を追加しない。
  - _Requirements: 2.5, 2.10, 7.11, 7.12, 7.13, 7.16_

- [x] 3. AtCoder 上の表示導線と問題トグルを差し込む
- [x] 3.1 ログイン判定と対象 DOM の解決条件を固定する
  - 確認済みヘッダー DOM シグナルだけでログイン状態を判定し、判定不能時は未ログインとして扱う。
  - MVP の対応 DOM を、確認済みの競技ページとトップページのヘッダー構造に限定し、それ以外では推測的に広げない。
  - ログイン判定や差し込み位置を解決できない場合は、そのページの導線表示を静かに停止する。
  - _Requirements: 2.1, 2.2, 2.3, 2.8, 7.11, 7.12, 7.15_

- [x] 3.2 ユーザーメニューの常設リンクを差し込む
  - ログイン時のみユーザーメニューへ「ac-revisit 操作」を `li > a` 構造で追加し、候補がなくても常に表示を試みる。
  - 既存の設定系メニューに近い位置へ優先して差し込み、メニュー末尾への無差別な追記を第一選択にしない。
  - 既存 UI に馴染む余白、行高、文字サイズ、小さな補助アイコンを維持し、独立パネルや浮遊ボタンを追加しない。
  - 挿入位置を解決できないページでは継続監視や再試行をせず、その場で省略する。
  - _Requirements: 2.3, 5.1, 5.2, 5.6, 5.7, 5.8, 5.9, 5.10, 7.12, 7.15_

- [x] 3.3 問題ページと提出詳細ページへ復習トグルを差し込む
  - 問題ページでは見出し内の解説ボタン直後または見出し末尾、提出詳細ページでは「問題」行のリンク直後を差し込み位置として使う。
  - 問題文脈または挿入位置を解決できない場合は表示しないまま止め、提出番号見出し横のような非対象位置へは出さない。
  - トグル文言は未登録時 `ac-revisit 追加`、登録済み時 `ac-revisit 解除` の 2 状態に固定し、AtCoder 既存の小型ボタンに近い見た目で状態色を分ける。
  - _Requirements: 2.2, 3.1, 3.2, 3.6, 3.7, 3.8, 3.9, 3.10, 7.12, 7.15_

- [x] 4. 復習ドメインの状態遷移を成立させる
- [x] 4.1 登録と解除を今日の一問状態と整合する単一更新で扱う
  - 未登録の問題を追加したときは当日を登録日として保存し、登録済みの問題を解除したときは復習対象から完全に削除する。
  - 今日の提案中の問題を解除した場合は、問題削除と今日の一問の完了状態への遷移をひとつの保存更新で確定する。
  - 自動 AC 検知や暗黙の完了判定を持ち込まず、利用者の明示操作だけを状態変更の根拠にする。
  - _Requirements: 3.3, 3.4, 3.5, 7.1, 7.9, 7.10_

- [x] 4.2 当日最初の提案処理で今日の一問を確定する
  - その暦日で最初に提案処理が起動した時だけ日次評価を消費し、候補があればランダムに 1 問を選んで当日の提案として設定する。
  - 前日未完了の提案は持ち越さず、候補があれば置き換え、候補がなければ今日の提案問題を持たない状態へ更新する。
  - 常に同時に扱う提案問題は 1 問だけに保ち、状態は「今日の一問が完了か未完了か」だけで管理する。
  - _Requirements: 4.3, 4.5, 4.6, 4.7_

- [x] 4.3 完了と「もう一問」の更新ルールを分離して実装する
  - 「完了」は提案中の問題を削除したうえで当日登録として再登録し、今日の一問を完了済みへ遷移させる。
  - 「もう一問」は今日の一問が完了済みのときだけ使えるようにし、未完了の問題を差し替える用途には使わせない。
  - 「もう一問」実行時は候補から新しい 1 問を選び、今日の一問を未完了へ戻す。候補がない場合は文言を保ったまま押下不可とする前提を維持する。
  - _Requirements: 6.5, 6.6, 6.7, 6.9, 6.10, 7.10_

- [x] 5. ポップアップの表示モデルと操作体験を実装する
- [x] 5.1 ポップアップ表示用の状態集約と view model を構築する
  - 保存済み workspace と当日候補の有無から、問題リンクの有効性、補助文、主要ボタン文言、押下可否を一箇所で決定する。
  - 今日の提案が存在して未完了ならタイトルリンクを有効化し、それ以外では同じ領域を非活性テキストとして描き分ける。
  - ボタンは未完了時の「完了」、完了済みで候補ありの「もう一問」、完了済みで候補なしの disabled「もう一問」、提案なし・候補なしの disabled「完了」を返せるようにする。
  - 補助文は当日の状態に応じて次の操作または 14 日ルールを説明する短文へ切り替える。
  - _Requirements: 6.1, 6.2, 6.4, 6.6, 6.7, 6.8, 6.17_

- [x] 5.2 共通モーダルとしてポップアップ UI を描画する
  - 常設リンク押下時と当日最初の自動通知時で同一 UI を使い、ヘッダー、本文、フッターの 3 領域を持つ一般的なモーダルとして表示する。
  - 本文では固定見出し「今日の一問」、補助文、問題タイトル領域、主要アクションを近接配置し、前景パネル、半透明 overlay、標準的なボタン表現を使う。
  - 狭い画面でも viewport 内に収まり、主要操作が横スクロールなしで読める幅制約を持たせる。
  - 閉じる操作はヘッダー右上とフッター補助操作の両方で提供し、一覧画面や追加通知面のような MVP 外 UI は増やさない。
  - _Requirements: 5.3, 5.4, 6.3, 6.13, 6.14, 6.15, 6.16, 7.2, 7.3, 7.7_

- [x] 5.3 ポップアップ操作前の整合確認と無言再同期を実装する
  - 問題タイトルリンクと押下可能ボタンの操作直前に、表示中の状態が操作時点の当日性と最新保存状態に一致するかを確認する。
  - stale 状態や日跨ぎを検出した場合は遷移や更新を進めず、必要に応じて当日の提案処理を伴う再同期後の最新状態へ再描画する。
  - ポップアップ表示中の継続監視や自動同期は行わず、再検証の契機を表示時と利用者の明示操作時に限定する。
  - _Requirements: 4.3, 5.3, 6.11, 6.12, 7.14_

- [x] 6. 起動順序と fail-closed 方針を統合する
- [x] 6.1 ページロード時の起動オーケストレーションを一本化する
  - ページロードごとに必要な依存を一度だけ組み立て、ログイン判定、導線差し込み、当日提案確定、自動通知判定の順で実行する。
  - 未ログインと判定したページでは、トグル、常設リンク、ポップアップ表示を行わず終了する。
  - すべての処理をブラウザ内だけで完結させ、外部 API や外部サーバーとの通信を追加しない。
  - _Requirements: 2.1, 2.3, 2.5, 2.8, 4.3, 5.4_

- [x] 6.2 自動通知条件と静かな停止動作を揃える
  - ページロード時の自動起動がその日の最初の提案処理だった場合に限り、自動表示通知を 1 日 1 回だけ許可する。
  - 候補が存在しない日は常設リンク以外の追加通知を出さず、前日の未完了提案も持ち越さない。
  - DOM 差し込み不能、問題文脈未解決、保存失敗は、利用者向けノイズを増やさず fail-closed または無言再描画で収束させる。
  - DOM 変更への対応は将来の追随修正で扱い、self-healing、継続監視、後方互換レイヤーは導入しない。
  - _Requirements: 2.10, 4.4, 5.5, 5.6, 7.11, 7.12, 7.13, 7.15_

- [x] 7. 実装を検証し、回帰を固定する
- [x] 7.1 (P) 日付計算、候補抽出、保存正規化の単体テストを整備する
  - 14 日判定、同日判定、候補抽出、ランダム選定の前提が設計どおりであることを固定する。
  - 保存データの空状態フォールバック、禁止フィールドの排除、保存失敗時の中止、last-write-wins 前提の扱いを検証する。
  - _Requirements: 2.4, 2.6, 2.7, 2.9, 2.10, 2.11, 4.1, 4.2, 4.6, 4.7, 7.4, 7.5, 7.8, 7.16_

- [x] 7.2 (P) 登録・解除・完了・再抽選の状態遷移テストを整備する
  - 登録、解除、提案中解除、完了、もう一問の各更新が単一スナップショットで期待どおりの状態を返すことを確認する。
  - 未完了時に「もう一問」を許可しないこと、候補なし時に提案を持たないこと、自己判断完了だけを根拠にすることを固定する。
  - _Requirements: 3.3, 3.4, 3.5, 4.3, 4.5, 6.5, 6.9, 6.10, 7.1, 7.6, 7.9, 7.10_

- [x] 7.3 (P) DOM 導線、ポップアップ、配布契約の統合テストを整備する
  - ログイン時と未ログイン時の導線差、メニューリンクの差し込み位置、問題ページと提出詳細ページのトグル位置と固定文言を確認する。
  - 常設リンクと自動通知が同一 UI を使うこと、ボタン状態、補助文、非活性表示、閉じる導線、無言再同期の挙動を検証する。
  - userscript metadata、最小権限、verify/build 契約、対象外 DOM で静かに停止する挙動を回帰対象に含める。
  - _Requirements: 1.1, 1.2, 1.4, 1.5, 2.1, 2.2, 2.3, 3.1, 3.2, 3.6, 3.7, 3.8, 3.9, 3.10, 4.4, 5.1, 5.2, 5.3, 5.4, 5.6, 5.7, 5.8, 5.9, 5.10, 6.1, 6.2, 6.3, 6.4, 6.6, 6.7, 6.8, 6.11, 6.12, 6.13, 6.14, 6.15, 6.16, 6.17, 7.11, 7.12, 7.14, 7.15_
